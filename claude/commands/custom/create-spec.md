# create-spec

入口 command：从模糊任务描述出发，访谈用户对齐**最终产物（L1）+ 用户视角 verify（L2）+ 横切取舍偏好**，写出可被 create-plan / implementer 当输入读的 spec.md。

spec.md 是面向使用者的**交付契约**——使用者拿它来判断"东西做对了吗"，不含任何实现取舍 / architecture / file-level decision。后续 create-plan 阶段在这份契约的框定下做 L3 设计。

## 何时使用

- 任务 L1/L2 规则重、需要在进入实现规划之前先把"成功画面"跟用户敲死
- 同一个任务后续会跑 create-plan，希望让 plan 阶段更聚焦 L3

## 什么时候**不**用

- 任务 L2 规则很轻（一两条 acceptance 一句话能写完）：直接用 create-plan，§2 facets 自带 L1/L2 对齐能力
- 单文件改动 / 小 bug fix：交付契约一句话能讲清，没必要单起 spec
- 已有手写 spec / PRD：直接喂给 create-plan 当输入即可

## 参数

| 参数 | 必需 | 说明 |
|---|---|---|
| task description | ✓ | 模糊任务描述（自由文本）。最终产物形态、使用者、范围等 alignment 信息可一并写入 |

## 1. Framing：交付契约 + 横切取舍

### 你产出什么、谁来用

交付物 `spec.md`，**两个读者**：

- **用户**：审 spec 是不是真的描述了"我想要的东西 + 我接受的样子"——这是交付契约的最后一道人审 gate
- **下游 create-plan / implementer session**：把 spec 当输入读，在这份契约框定下做 L3 设计 / 实现

**spec.md 的内容边界**：

| 含 | 不含 |
|---|---|
| Layer 1 (L1)：最终产物形态 + 使用方式（使用者拿它做什么决策 / 后续动作） | architecture / 管线分层 / 接口契约 |
| 横切：取舍偏好（L1+L2+L3 三层影响，但 L3 影响只**列出**不**决策**——决策权给 plan 阶段） | 内部 verify / types / lint / 单元测试 |
| Layer 2 (L2)：用户视角 verify（独立于内部实现的『算交付完成』可观测条件） | file-level 修改清单 |
| 假设、范围、硬限制（合规 / 性能 / 成本 / 兼容性 / 安全） | 错误处理 / 重试逻辑 / 日志策略 |

如果你发现自己开始谈"用什么管线 / 哪些文件 / 接口长什么样"——那是 L3，停下来留给 create-plan。

### 三层产物意识：spec 是 L1+L2 的载体

**Layer 1 (L1) — 最终产物 + 使用方式**：implementer 执行完整套流程后交给『真实使用者』的东西，**以及使用者拿到它用来做什么决策 / 后续动作**。"是什么"和"用来做什么"同等重要——同样物理形式的产物（一份报告 / 一个 API / 一个 CLI），用途不同会彻底改变它需要的分析深度和评价指标。例如同样是"对比两个 AIGC GIF 系统的报告"，用途如果是"选一个上线"，黑盒分数足够支撑决策；用途如果是"取长补短做一个新的"，则必须深入到算法/模块层让用户能迁移其中的局部优势。

**Layer 2 (L2) — 用户视角 verify**（提前定义，TDD-style）：从 L1 使用者视角看『算交付完成』的可观测条件。这层**独立于内部实现**——使用者不关心你怎么做，只关心产物在他们眼里是不是 work。因此可以、也应该在 plan 阶段做实现取舍**之前**先收敛。类比 TDD 的 RED：先写"成功长什么样"，再去想怎么实现。

**Layer 3 (L3) — 实现取舍 + 内部 verify**：管线分层、接口契约、可观测性、错误处理、单元测试、关键不变式断言等。**不属于 spec**，由下游 create-plan 在 L1+L2 框定下做。

### 横切：取舍偏好（贯穿 L1 + L2 + L3）

取舍偏好不属于任何一层，而是**贯穿三层的横切关注点**——用户在多个维度之间的相对优先级，会在每一层触发 multi-option 决策：

- **在 L1**：影响产物形态选择（"傻瓜式一键交付 vs 高级可调界面"——选哪个会改变最终产物本身）
- **在 L2**：影响 verify 维度优先级和阈值（"步骤少 + 时延达标 vs 最终效果质量上限"——验证哪些维度、阈值多严）
- **在 L3**：影响实现取舍（"代码复杂度 vs 可演进性"、"性能 vs 稳定性"）——**spec 只列出 L3 会被影响的方向，不替 plan 决策**

**关键认知**：用户对任务的初始描述**通常不会包含这些权衡**——它们是隐含的、需要 spec 阶段显式跟用户对齐。AIGC / UX-heavy 产品尤其常见；纯 binary feature（有或没有）任务可省略此步。

**spec 是取舍偏好的主谈场**——一旦在 spec 里跟用户拍下，下游 create-plan 直接读 spec 而不再追问；plan 阶段重新发现的取舍漂移属于实施现场推翻 spec 契约，按 long-task-protocol §6 处理。

### 关于顺序

建议访谈顺序：**L1 → 取舍偏好 → L2**。L1 收敛产物形态后立刻讨论取舍，再用对齐过的取舍框定 L2 verify 维度。但**允许迭代回退**——L2 verify 写到一半发现新维度漏了取舍，回头补；写 L1 时发现使用者形态影响取舍，回头补。**严格顺序不重要，反推方向 + 横切意识重要**。

### spec.md 的位置

落点 + 下游 create-plan 如何读：见 §3「输出：spec.md」。

### 风格与取舍

遵循 `~/.claude/references/deep-discuss-style.md`——spec 阶段质量 ≫ 速度，访谈可以更长、用户介入更高频。

## 2. 需要对齐的点（不限于此）

spec author 和 user 对齐的过程中，至少要让以下几类信息变清晰。**这不是顺序步骤**——可以并行、迭代、回头补；**也不是穷举清单**——任务特性需要的其他对齐点（compliance / 隐私 / 国际化 等）随时加入。

通用 lens：**"为了让下游 create-plan / implementer 能在『使用者契约』层面无歧义，我现在缺哪些只能用户回答的信息？"** 剩余决策都能被 agent 合理 default 时，对齐充分。

研究 / 探索 / 跑命令确认假设是对齐的有机组成部分——读 reference 代码、查外部规范、跑 probe 确认环境——能让你给出更高质量 spec 的动作都该做，不要因为没明确写出就跳过。

**borderline 决策的两条 path**：

- **反转成本高**（一旦写进 spec、plan/implementer 跟下去再翻盘代价大）→ 立刻 AskUserQuestion："决策 X / 我的 default 是 Y / 理由 Z / 你确认还是推翻？"
- **能合理 default**（reviewer 看了能审、翻盘成本低）→ spec author 自己拍 + 写进 spec 末 Defaulted Decision 表给 reviewer 审

### 最终产物 + 使用形态 + 使用方式（L1）

**对齐**：spec 落地后给『真实使用者』的产物形态、使用形态（怎么调用 / 在哪运行）、**使用方式（拿到产物用来做什么决策 / 后续动作）**。这是一切设计的反推起点。

**lens**：使用者是谁、怎么调用、**拿到产物用来做什么**？同形式不同用途下需要的设计深度可以差好几个数量级（例见 §1.2 L1 段）。什么明确算成功？这个定义往往主观，取决于使用者心中的使用场景，需要主动挖出来。哪些约束 / 假设是硬限制？spec author 在做哪些隐式假设、需要用户确认？

**常见询问方向锚点**（不限于此，按 lens 展开）：

- **形态 + 调用 + 边界**：终端用户 / lib / API / 系统组件 / 部署单元——一类还是多类；同步 / 异步 / 流式 / 事件触发；本地脚本到分布式集群之间哪一档
- **使用方式**：使用者拿到产物**用来做什么决策 / 后续动作**——这一项决定下游设计深度（同形式不同用途差好几个数量级）
- **成功定义**：使用者心中的成功画面（往往主观，需主动挖；"质量达标 / 用户可用 / 完成"指什么）
- **范围 + 硬限制 + 隐式假设**：明确**不**做什么、合规/性能/成本/兼容性/安全边界、spec author 识别但需用户确认的假设

**为什么这点是反推起点**：跳过它直接谈取舍最常见的失败是**过度设计**（设计了根本不需要的层）或**遗漏设计**（漏掉真正需要详细设计的层）。L1 错了，下游 L2 verify 全错、plan 阶段的 L3 取舍维度也全错——所以这里必须先收敛。

### 取舍偏好（横切 L1 + L2 + L3）

**对齐**：用户在多个维度之间的相对优先级。横切影响（L1/L2/L3 三层各塑造什么）+ 例子 + 适用范围见 §1「横切：取舍偏好」；本节聚焦**访谈中怎么挖出来 + 怎么让用户拍**。

**lens**：这次任务的关键维度是哪几个？用户更看重哪些、愿意牺牲哪些？哪些权衡是用户初始描述**没显式提到**、需要 spec author 主动摆候选让用户拍板的？

**关键动作**：spec author 识别需要拍板的取舍 → 梳理候选维度 + pros / cons → 用 `AskUserQuestion` 让用户拍板（不是 prose 里列方案让用户回复）。决策权在用户，spec author 的职责是把选项摆出来。

**spec 里只列 L3 会被影响的方向，不替 plan 决策**——例如"用户优先稳定性"是 L1+L2 已确定的偏好，但 plan 阶段"具体用哪种 retry 策略"是 L3 的事，spec 不替它拍。

### 用户视角 verify（L2）

**对齐**：从 L1 使用者视角看『算交付完成』的可观测条件。这层独立于内部实现，应在 plan 阶段做设计取舍之前先收敛——类比 TDD 的 RED：先写"成功的样子"，再去想怎么实现。

**前置依赖**：L2 verify 的内容由 L1（产物 + 使用方式）决定；维度优先级和阈值由取舍偏好决定。所以本节前应已对齐取舍偏好（见上一节）；如果到这里发现 verify 维度该测什么不清楚，回头补取舍偏好。

**lens**：把自己代入使用者——产物到手后，使用者会用什么 workflow 检查它 work？这些检查的可观测形式是什么（命令输出 / 文件内容 / UI 截图 / 评测分数 / 人工判断 / 跨多步业务流）？哪些是 happy path、哪些是 error / 边界路径？

写不出 verify、还要等"实现完了再说"——那是 L1 没收敛清楚，回头补 L1。

**常见询问方向**（不限于此）：

- **happy path verify**：使用者怎么验证功能成功？跑命令看输出 / 点 UI 看反馈 / 读文件看内容 / 跑评测看分数 / 跨多步业务流？
- **error / 边界 verify**：设计上要兜的失败模式（超时 / 输入异常 / 权限缺失 / 性能不达标）哪些必须可被使用者验证？
- **verify 的形式不强求 shell**：使用者接受的不一定是"命令 + stdout"——视觉判断 / 主观质量 / 人工对照都是合法形态，不要硬塞成 shell 输出
- **agent 自动 vs 人工介入**：哪些 verify 步骤 agent 可独立完成、哪些必须人工（真机 / 真账号 / 主观判断）？人工 gate 前应用 agent 自动检查兜底主要风险，压低人工 gate 失败概率

**关键**：L2 必须独立于内部实现——既不写"内部数据结构 X" / "状态机 Y" 这类内部术语（那是 L3），也不从代码内部状态机 / pipeline 反推 verify 维度（尤其多阶段 / 多状态 / 多页面产品，要按使用者实际经历的路径定）。

## 3. 输出：spec.md

落点：`./plans/<YYYYMMDD>-<short-name>/spec.md`
- `<short-name>`：从任务描述提 3-5 词 kebab-case
- 同任务后续 plan.md 落同一子目录（`./plans/<YYYYMMDD>-<short-name>/plan.md`）
- 同日重名：子目录加 `_v2`、`_v3` 后缀
- 子目录不存在则新建

格式自由，结构跟着任务走。下表是 writer 写完后的自检清单（必答项粒度）；完整 reviewer gate 见 `~/.claude/references/spec-review-principles.md` + `/custom:review-spec`。

| spec 必须能答出 | 不合格示例 |
|---|---|
| **最终产物形态** + **使用者** + **使用方式**（拿到产物做什么决策 / 后续动作） | "做一个 X" / 没说使用者 / 没说用来干嘛 |
| **范围 / 约束 / 假设**（明确**不**做什么；硬限制；spec author 隐式假设需用户确认的） | "差不多就行" / 没列边界 |
| **取舍偏好 + 三层影响**：用户在产品 / UX / 工程维度上的相对优先级，以及它在 L1 产物形态、L2 verify 维度阈值、L3 实现方向各自塑造了什么；不适用任务（binary feature 等）需说明原因 | "随便选" / 没列 / 只列一句"用户说要好用" |
| **用户视角 verify**（L2：从最终产物使用者视角看『算交付完成』的可观测条件，覆盖 happy + error path；可执行 = 命令 / 截图比对 / 评测分数 / 人工判断都行，但必须独立于内部实现） | "跑一下应该 OK" / "类型对了就行" / 写到内部数据结构 |
| **verify 步骤的人机边界**：每条 verify 标识 agent 可独立完成 vs 需人工介入；人工项必须说明自动化先兜底了哪些 | 没标 → plan/implementer 不知道哪步会被 block |
| 哪些决策是访谈中没问、spec author 自己拍的（"我默认 X，因为 Y"） | 没列 → reviewer 无从审 |

**spec 不写**（这些归 plan）：
- architecture / 管线分层 / 接口契约 / file-level 修改清单
- 内部 verify（types / lint / unit test / contract test）
- 错误处理 / 重试 / 日志策略
- **"Surface-to-plan" sidebars / reminders / 兜底段落** —— 判定规则见 `~/.claude/references/spec-review-principles.md` §4 information-symmetry lens

如果发现自己想写这些——记下来留给 create-plan 阶段，spec 里只放使用者视角看到的部分。

### 不留 Open Question：升级路径

spec **不留 open question**。任何 OQ 必须先走升级：

1. **能 research 的** → spec author 自查（读代码、查规范、跑命令 verify）
2. **research 也答不出的** → 当场用 AskUserQuestion 问用户
3. **用户明确同意 default X** → 记为 Defaulted Decision，不留 OQ
4. **用户答"后面再说"或回答模糊** → 是搁置不是授权。给出具体 default + 一句理由，再问一次 Y/N 确认；确认后再记 Defaulted Decision

允许在 spec 里留：

- **assumption**：spec author 显式列出的、可能影响契约理解的假设（标记 "若错请纠正"）

不要为了"看起来完整"而幻觉细节，但也不要用 OQ 替代调研。

### 审查

自检通过后、handoff 之前，执行 `/custom:review-spec <spec path>` 进行循环审查。循环未终止不进入 handoff——review-spec 的 gate 比 §3 表格更全，跳过会让下游 plan / implementer 撞上未发现的契约缺陷。

- **收敛性**：判断 finding 是否需修。需修 → 改 → 重审。循环到一轮无需修。

### Handoff

写完且审查落地后打印：

```
spec written: /abs/path/to/plans/<date>-<name>/spec.md

下一步：
  - 进入 plan 阶段：在新 session 里跑 `claude '/create-plan 把这份 spec 转成 plan：<spec path>'`
  - 或直接审 spec：跟用户走过 spec 内容、确认契约符合预期
```

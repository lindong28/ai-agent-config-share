# Plan Execution Principles

执行 plan 时关于"该不该停下来"的判断原则。

**默认姿态**：executor 应最大化独立完成 plan 内容，只在必要时 stop。停止或等待外部支持显著增加人力成本和等待时间，减少不必要的 stop 是核心目标。

**怎么用**：每次"我是不是该停了"的念头，按 1 → 2 → 3 → 4 自检——任何一关没通过都不该 stop。

---

## 0. Stop Gate（停止前必须逐项通过）

以任何理由结束当前执行的 final response，都算 stop。状态汇报后不继续执行，也算 stop。

Stop 前必须逐项满足：

1. **必要性已证明**：已排除排队等待、应用层问题、适配层问题；不能确认时 default-to-continue（自己继续等，**同时发一条消息让用户看到当前归因状态 + 触发 stop 的条件**）。
2. **归因已分层**：已检查应用层 / 适配层 / 外部服务层 / 人工层（**人工层 ≠ Web UI；先试 browser automation / DevTools / Computer Use**）；把剩余步骤归为人工层前，必须证明 executor 已经推进到**不可代办的交互边界**（认证确认、MFA、授权、主观裁量、高风险确认等），而不是停在 CLI / wrapper / 文档表层报错；只有观察过第三方原始响应、standalone probe、状态页、日志或工单，才能归因为外部失败。
3. **替代路径已尝试**：已尝试可用的 API / DB / CLI / DevTools / browser / local script / mock / direct probe 等横向替代路径，以及对当前路径的纵向缓解；"有外部失败的直接证据" 只证明当前路径失败，不证明所有 executor 可控缓解已穷尽。若某条路径被 wrapper / adapter 报错挡住，必须向更接近原始能力的层（local endpoint / direct probe / GUI automation）再探一层，再升级为"不可达"。
4. **verify 已拆分**：已把 executor 能独立验证的子部分做完（**`人工 / UX / 审美` 标签 ≠ 100% 用户裁量**——日期格式可读、文字截断、布局明显错乱、链接 404 等 common-sense 通用判断也属 executor 可独立验证）；剩余部分确实必须用户介入或依赖不可用能力。
5. **交接可执行**：final response 明确为什么停、阻塞哪一步、已覆盖什么、用户需要做什么；用户不需要追问或自行研究即可执行。若需要用户交互，executor 必须先生成或打开可执行交互物，并说明完成后 executor 将如何继续。

## 1. 只在必要时 Stop

**判定"必要"前必须排除两类伪必要：(a) 排队等待——交给用户也只能继续等；(b) 应用层问题伪装成外部失败——agent 自己能修。归因不确定时 default-to-continue，不 stop。**

### 分层归因

遇到"看起来 blocked"时先问——故障可能在哪一层？

| 层 | 检查角度 |
|---|---|
| 应用层 | 自己的状态机、解析、缓存、轮询是否漏掉了已有结果 |
| 适配层 | SDK / wrapper、参数、认证、project / session 语义是否对 |
| 外部服务层 | 第三方原始 API / job 是否真的没产出 |
| 人工层 | 是否真的需要用户本人完成认证 / 授权 / MFA / 主观判断 / 高风险确认 |

人工层不是"需要打开网页"的同义词。只要 executor 有 browser / DevTools / Computer Use 能力，Web UI 也是可执行路径；只有观察到认证、权限、主观判断或高风险确认等直接 blocker 后，才能把剩余步骤归为人工层。

只有**直接观察过外部服务层**才能写"外部不可用"——第三方原始响应、状态页、standalone probe、服务方日志/工单之一。"我们应用没拿到结果"只是"当前路径未产出"，不能升级为外部失败。

### 等待场景下 default-to-continue

排队等待场景（视频/图像生成、异步 job、有 `pending/generating` 但无 ETA 的服务）下，stop 交给用户也只能继续等——因此 executor 应自己继续等，前台等待时把不依赖该外部产物的工作做掉（清理诊断代码、跑非依赖测试、更新 state/journal）。

归因不确定（既不能确证外部失败，也不能确证自己能修）时同样走这条路径：自己继续等，**同时发一条消息让用户在最新消息里看到**。消息走简化契约——不要求等于 stop 信息，但必须让用户能验证 executor 的判定逻辑：

| 字段 | 内容 |
|---|---|
| 为什么还在等 | 当前归因状态、为什么没升级为 stop |
| 什么会让我 stop | 触发 stop 的条件（拿到外部失败的直接证据 / 等待预算耗尽且无 yield 路径 / 用户介入要求） |

default-to-continue 不等于"用同一种方式无限等 / 无限重跑"。等待中若发现自己反复重启同一类完整任务、或同阶段反复失败，转 §2 时机拆分，不要继续端到端重跑。

### 反模式

- 没看第三方原始响应就断言外部失败。
- 没试过 API / DB / DevTools / Computer Use 等替代路径就要求用户手动操作。
- CLI / API 路径失败后，未尝试可用的 Web 控制台 browser automation，就把后台页面操作交给用户。
- 把"我等够久了"当作外部失败的直接证据。

## 2. Verify 拆分（stop 前 / 执行中）

**Verify 步骤通常不是原子的——不论是 stop 前的人工交接，还是执行中长时间不收敛的循环，都应先按维度、链路或时机拆分，把 executor 能独立验证的子部分做掉，再决定要不要把剩余推给用户或重跑完整链路。**

人工 verify 有更高的"一次成功率"，前提是 executor 已把可自动覆盖的风险压缩干净。整体推给用户或盲目端到端重跑 = 浪费人力 + 把多变量混在一起难以归因。

### 维度 / 链路拆分（stop 前）

| 拆分方式 | 例 |
|---|---|
| **维度拆分** | 产物拆三类——客观可测（响应码 / 字段 / 布局溢出）、**common-sense 通用判断**（日期可读、文字截断、布局明显错乱、链接 404）、主观偏好（品牌调性、个人审美阈值）——前两类 executor 先做完，最后一类交用户。**`人工 / UX / 审美` 标签 ≠ 100% 用户裁量**；不要把"是否满意 / 是否接受"伪装成技术 blocker。 |
| **链路拆分** | 最终需用户手机跑小程序验证，但可在服务器侧模拟用户输入跑后端生成图片 → 先验证后端链路质量 |

拆分原则：每个子部分要么 executor 能独立验证、要么必须用户介入——尽量把前者做完，把后者写清。

### 时机拆分（执行中也要拆）

verify 拆分不只用于 stop 前。当同一 verify gate 长时间不收敛、或同阶段反复失败时，应当场把完整 E2E 拆成阶段化验证：固定已通过阶段的中间产物，把 prompt / 输入 / 子链路单独验证后再串联。

用完整 E2E 调试单一阶段会把多个变量混在一起，单次反馈周期通常也远超阶段反馈。

### 反模式

- 只看 verify 步骤描述、不思考它是否可拆分，整体推给用户。
- 拆分后跳过 executor 可做的子部分，直接列出要用户做的事。
- UX / 审美验收整体推给用户，没先做 common-sense 通用判断（日期可读、文字截断、布局明显错乱）。
- 已经有合格中间产物，仍为了下游小改动从头重跑完整链路。
- 用最终端到端失败来调前置阶段，导致变量混在一起。

## 3. 不要把 sample-pass 当作 done / stop 信号

**一次 verify PASS 如果依赖大量重抽、单项 retry、或 executor 手工把模拟用户输入改成更专业的版本，这只是 sample-pass——不能据此 stop、也不能标记为 done。修复必须落到所有用户都能受益的可复用层（prompt / schema / scorer / 默认参数），把洞察转成可复用修复后重测。**

### 可复用 vs 一次性

| 类别 | 例 |
|---|---|
| 可复用上游修复 | meta prompt、schema 字段、质量评分、候选排序、默认参数调整 |
| 局部下游补救 | 单项 retry、复用已生成中间产物（冒充最终产物，详见 §4）、转码 fallback |
| 测试绕路 | executor 手工专业化用户输入、抽到满意为止但不改系统、绕过流程修数据 |

如果成功依赖测试绕路，必须记为 iteration——保留失败输入与坏产物作为 iteration 证据，把洞察转成可复用修复后重测，不能直接计为 done。

### 反模式

- 把"agent 多试几次终于成功"等同于"用户体验已经达标"。
- 同一类质量问题只增加 retry 次数，不改 prompt / schema / 质量门。

## 4. Verify 判据与产物的诚实性

**一次 PASS 要成立，"判据没被偷改 + 产物是真的"两条同时满足。判据被降级、或用历史/中间产物冒充最终产物，都让 PASS 变假——不算 done、也不算 stop 依据。本节与 §3 同族：§3 是"达标靠绕路"，本节是"判据本身或产物本身不诚实"。**

### 判据不可降级

plan / spec / contract 写下的每条判据就是 gate 上要守的线，executor 不可在执行中单方面放松。任何把原判据替换成更容易满足的代理判据都算降级——下面是常见形态，非穷举：

| 降级形态 | 例 |
|---|---|
| 终态降成中段态 | "产物可见 / 可下载 / 状态 done" 降成 "任务已提交 / 进入生成中页面" |
| 强验证降成弱验证 | "端到端真实产出" 降成 "代码路径存在 / 单元测试通过" |
| 硬指标降成软指标 | "延迟 < 200ms" 降成 "看起来够快" |

要降级必须先停下来让用户裁定（AskUserQuestion）；获批后在判据的权威来源处留 audit trail——有 plan / spec / contract 则改文件，无书面来源（ad-hoc 任务）则把新判据回写给用户确认。绕过这条 = 整个验收失去基准，所有下游 PASS 都不可信。

### 产物不可替代

PASS 的证据必须来自**本次执行链路的真实产出**，不能用历史 / 缓存 / 上一轮 session 的结果顶替——无论产出是 tangible 产物（导出文件、生成的图 / 视频 / GIF、落库记录、状态变更）还是纯推理结论。这是 §3"复用已生成中间产物"的产物侧对应面。

**判据 lens**：列出这条 step 的证据问自己——"删掉本次执行之前的所有历史数据，这条证据还成立吗？"答"不成立"就不是本次真实产出。

tangible 产物还有两种典型冒充：链路中段态当终态（"任务已提交" ≠ "产物已生成"）；异步链路没等到就标 PASS——即便等 30+ 分钟也必须等到真实产物，或把"等到产物"列为 open issue 显式留下。

### 反模式

- 写 fix / re-test / 子任务 prompt 时把判据写松（"只要进入 X 页即可"），没经用户批准。

## 5. Stop 时给清晰可执行的交接信息

**Stop 是契约动作，不是放弃动作。停时必须让用户能 (a) 验证 executor 的 stop 决策合理；(b) 高效率执行所需行动，不需要回头查文档或追问。**

Ask yourself：用户读完这段不再做任何研究 / 追问，能否同时判定 stop 是否合理 + 直接照做？任一为否就回去补。

### 信息完整性

停时的最终回复必须明确：

| 字段 | 内容 |
|---|---|
| 为什么停 | 直接证据级别的阻塞原因，不是推测 |
| 阻塞了哪步 | 哪个 verify 步骤 / done gate（具体到 task id 或 verify 步骤名）|
| 已独立覆盖什么 | 原则 2 拆分后 executor 已通过的子部分；无可独立部分时写明"无可拆分子部分" |
| 用户需要做什么 | 明确动作 |

### 可执行性

用户行动指令要假设最少的用户知识——避免用户回头查文档或追问怎么做：

- 需要扫码 → 直接给二维码图片
- 需要登录网站 → 给链接 + 访问路径，不要"去某某后台找一下"
- 需要触发流程 → 给具体步骤，不要"重现一下那个 bug"

### 反模式

- "blocked，请用户介入" 一句话停止，没说阻塞了什么 verify。
- 把"可能 blocker"写成"确认 blocked"，让用户做无谓介入。
- 给出需要用户进一步研究/查询才能执行的 instruction。

## 6. 执行中发现的问题应被记录

**执行中发现的问题——不论 plan / long-task / supervise / 一次性 task / 探索——只要不在本次执行的修复范围内、本次又不打算就地修，就必须在执行结束前落到一个 issue 文件，而不是只活在你这次的 context 里。**

你交付完，context 就消失了。问题如果只活在本次执行的对话里，未来的 agent（甚至 resume 的你自己）只能重新踩坑——而踩坑的成本通常远高于"当时多写两行 issue"。

### 什么算"问题"

本次范围内**会修的**事不算（那是 task 本身）；范围外**值得未来某次修的**事算。下面是常见形态，非穷举：

| 形态 | 例 |
|---|---|
| 产品 bug / reliability 缺口 | 你修的是 A，但路上看到 B 也有问题；外部 API 偶发失败的频率、影响 |
| 工具链 / 适配层异常 | CLI 偶发 hang / wrapper 行为异常 / 第三方 SDK 反复出现的怪行为 |
| Agent 行为问题 / skill 缺口 | supervise 场景下 wrapped agent 反复犯同类错 |
| 文档错误 / convention 违反 | 现有文档与实际行为不符 |

**覆盖 ≠ 持续追踪**：「contract / 流程已经覆盖了这个行为」不等于「不需要记录」——前端 UX 覆盖 ≠ 后端 reliability 维度被持续追踪；前者是规则，后者是数据。

### 落点（WHERE）

按落点 lens 决定：
- 本次 task 内会回头处理的 → `state.md` Open Issues（task 内 lifecycle）
- 项目级、跨 session 长期跟踪的 → `docs/issues/<domain>.md`（按 domain 分文件）
- 两者都没有的一次性 task / 探索 → 写进交付时的 final response / handoff（该场景下唯一必在的落点）

### 反模式

- handoff / commit message 里口头提到「顺便发现 X」但没写入任何 issue 文件——下次没人找得到。
- 把问题推到「下一次 task 再处理」——下一次的 agent 不一定读得到本次 context。
- 只在被人问起时才记——主动义务不靠 prompt 触发。

## 与 long-task-protocol 的关系

| 关注点 | 由谁规定 |
|---|---|
| 状态外部化（state.md / journal.md）、交付前 verify | `long-task-protocol.md` |
| 偏离处理、硬 blocker 找用户修根因 | `long-task-protocol.md` §6–§7 |
| **Stop 必要性 gate / Verify 拆分（stop 前 + 执行中）/ 成功定义 / 判据与产物诚实性 / 交接信息 / 发现的问题应被记录** | 本文件 |

本文件的原则（Stop Gate、Verify 拆分、判据与产物诚实性、交接信息、发现即记录）不限于 plan 执行场景或 long-task 模式。

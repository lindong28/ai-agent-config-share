# Game UX Verification Principles

游戏（**玩的过程本身即价值**）的**游戏体验验收**原则。补充顶层 `ux-contract-review-principles.md`：那份验"功能对不对"，这份验"体验好不好"。

**何时加载**：仅当 ux-contract 在 L1 声明**产品类型为游戏**时。
- `create-ux-contract`：用每条的 **create lens** 现场对齐意图、写「游戏体验验收」段。
- `review-ux-contract`：用每条的 **review flag lens** 审「游戏体验验收」段是否到位。
- `execute-ux-contract`：按每条的**证据类型**在 playthrough 中采证、对照 bar 判断。

**为什么单独成文**：功能验收是二元断言（做 X → 观测 Y），顶层 `ux-contract-review-principles.md` 第 2 条甚至要 flag "没有可观测 pass/fail 的主观质量"。但体验是**整局的形状**——比率、密度、成本——不是某个时点能断言的状态，只能 **playthrough 测量 + 对照"意图中的体验"判断**（judgment + evidence）。bar 是 **intent-relative** 的：好不好玩相对于**这款产品意图中的乐趣**，不是某个普适数字。已有先例——叙事质量 / 视觉美学也写"judge / 人工审"，本文件把同一姿态用到**玩的循环本身**。

**现场定义为主**：什么算趣味、什么算 grind、可容忍多少——由 create 阶段向用户**现场问出**，针对每款产品定义。**先判方向再判量**：G1/G2 默认把摩擦 / 重复 / grind 当负向成本压低，但受苦系 / idle / 刷宝类恰以其本身为乐趣——create 时先确认对本产品它是成本还是乐趣，是乐趣则两条反向读（不再压低，转而测它带来的乐趣够不够）。每条原则附一个**起手提示清单**，仅作卡住时的 memory-jogger，**不是终止清单**（见反模式）。

---

## G0. 规格锚定（Spec Anchoring）

**游戏的体验验收只有"对照设计意图"才成立。验收目标始终是用户拍板的设计意图——spec（设计文档 / GDD / 游戏介绍）是它最好的书面记录，半成品 build 不是替代；故契约锚定 spec，不纯从观测 build 反推。** spec 至少覆盖：**核心玩法**、**为什么好玩**（乐趣 / 张力 / stakes 来源，含胜负 / 失败条件）、**怎么玩**。

**create 对齐 lens**：
- 开工前向用户**索取 spec**；**没提供就 AskUserQuestion 索取**，不纯从观测 build 反推契约。
- 以 spec 为 L1 + 验收侧重 + 体验验收段的锚，把核心要素写成可验项；乐趣轴（"为什么好玩"）的"可验"是判断 + 证据、不是二元阈值。
- **spec ↔ build 或 spec ↔ 现场定义背离时 surface，不静默选一边**：冲突时（例：spec 承诺"没飞升就老死"，build 寿命到上限却不终止游戏），写成 verify 项或 AskUserQuestion 让用户裁决哪边是当前意图，**禁止静默 Defaulted Decision**；拿不准默认 ask，别默认判 spec 赢。
- 用户**确实没有任何 spec** 时：允许继续，但在契约顶部标 `spec-unanchored：体验验收为意图盲`，把你从观测 + 访谈推断的核心玩法 / 为什么好玩 / 胜负条件**回写给用户确认**（把推断变成用户拍过的意图）。

**review flag lens**：
- 契约有没有覆盖 spec 的**核心要素**？还是只描述了 build 现状、没对照设计意图？
- 有没有 **Defaulted Decision 静默背离了 spec / 现场定义**而没 surface 成 verify 项 / 待裁决？
- 契约是 **spec-anchored 还是 spec-unanchored**？标了 unanchored 的，推断出的意图有没有回写给用户确认？

**审查自问**：契约够不够让 execute 拿设计意图当验收 ground-truth、而非把 build 现状当目标？不够就 flag——上面是常见缺口、非全集。

**证据 / 产出**：契约体验验收段能追溯到 spec 的哪条；spec ↔ build 背离清单（若有，各自标处置：verify 项 / 待裁决）。

## G1. 趣味密度 / 乏味占比

**核心趣味 beat 与无脑重复操作之间的间距，不能稀到让玩家在抵达下一个趣味点前就流失。**

**判断基准**（intent-relative）：对照 create 时对齐的"意图乐趣"——趣味 beat 的密度撑不撑得住中间的重复；是否存在玩家要熬过长段纯重复才够到下一个趣味点的"grind 墙"。**若 create 已判定本产品以重复本身为乐趣**，则反向判断：测这套重复本身带来的乐趣够不够，而非 grind 段够不够短。

**create 对齐 lens**（向用户问出，写进契约「游戏体验验收」段）：
- 这款产品意图中的核心循环是什么？
- 哪些算**趣味 beat**、哪些算 **grind**？（由用户针对本产品定义）
- 抵达下一个趣味点前，意图能容忍多少重复 / 多长 grind？早期容忍度应更低，还是早期的高重复 / 高门槛本身是有意为之（买断 / 艺术 / 受苦系）？
- 起手提示（卡住时）：常见趣味 beat——有意义抉择 / 新剧情或新机制揭示 / 资源拐点 / 关系推进；常见 grind——重复同一行动无新信息 / 纯数值刷取。

**review flag lens**：
- 契约有没有把"趣味 / grind 定义 + 容忍度"写成可判断的游戏体验验收项？
- 高价值趣味内容有没有"被 grind 淹没"的风险未被覆盖？
- 是否只写了"功能存在"（如"有事件系统"）却没写"密度够不够"？
- 有没有写明要采的**证据类型**？缺了 execute 无从采证、bar 无从对照。

**审查自问**：契约够不够让 execute 按本产品意图中的趣味密度，判断「趣味撑不撑得住中间的重复」？不够就 flag——上面是常见缺口、非全集。

**证据类型**（execute 在 playthrough 采集）：beat / grind 时间线（按操作序列标注）、最长连续 grind 段、单位操作趣味占比。

## G2. 交互摩擦成本

**抵达并执行一个核心交互单元所摊到的点击 / 滚动 / 阅读，不能高到让操作本身成为劝退点。**

**判断基准**（intent-relative）：对照 create 时对齐的"意图心流"——核心交互单元的交互成本是否在意图范围内；核心内容是否被埋在过多翻屏 / 导航之后。**若 create 已判定本产品以摩擦本身为乐趣**，则反向判断：测这份摩擦本身带来的乐趣够不够，而非交互成本够不够低。

**create 对齐 lens**：
- 这款产品的"**核心交互单元**"是什么？（不限于决策——创作 / 推进 / 观察 等也算，按产品定）
- 意图中完成它该有多顺（几步内、要不要翻屏 / 找入口）？
- 哪些信息是该交互必需、哪些是可折叠的噪声？
- 起手提示：到核心入口的步数 / 首屏是否直达核心 / 每个核心交互单元的必读字数 / 反馈链长度。

**review flag lens**：
- 契约有没有把"核心交互单元的交互成本上限 / 心流预期"写成可判断项？
- 是否只验了"可点 / 不溢出"而漏了"够不够顺、要不要划很多屏"？
- 是否只验了"入口 / CTA 在首屏内"，却漏了"行动后的反馈 / 结果在首屏内可见、且累积历史后仍可见"？
- 有没有写明要采的**证据类型**？缺了 execute 无从采证。

**审查自问**：契约够不够让 execute 按本产品意图中的交互心流，判断「核心交互单元的成本没把操作变成劝退点」？不够就 flag——上面是常见缺口、非全集。

**证据类型**：每个核心交互单元的点击数 / 滚动次数 / 必读字数、到核心入口的步数、首屏是否直达核心内容、行动后反馈元素（叙事 / 数值反馈）rect 是否落在默认视口内（无需滚动、累积历史下仍成立）——非只断言文本在 DOM 里存在（见 GP4）。

## 反模式

| 反模式 | 为什么不要 |
|---|---|
| 把体验 bar 写成 fake 阈值（"≤5 次点击"）当客观 gate | 体验是 intent-relative 判断；硬阈值易被 game、跨产品不可移植。用 judgment + evidence，量化只作证据不作 gate。 |
| 起手提示当勾选表逐项过 | 提示是 memory-jogger，不是终止清单；逐项打勾退回 dimension-driven，丢掉对本产品的针对性（goal-directed beats dimension-driven）。 |
| 验"功能存在"冒充验"体验" | "有事件系统"≠"趣味密度够"、"按钮可点"≠"交互够顺"；存在性不等于体验质量。 |

# Game UX Test Patterns

游戏测试时容易被通用 checklist 漏掉的**体验失败形状库**。补充顶层 `ux-test-patterns.md`（那份偏内容 / 资讯产品，P1–P10）。**仅当 ux-contract 声明产品类型为游戏时加载。** 每条描述"乐趣失败长什么样"——execute 在 playthrough 中据此识别。遇到新形状可新增。

对应游戏体验验收原则见 `game/ux-contract-review-principles.md`（G1 趣味密度 / G2 交互摩擦）。判断都要**对照契约「游戏体验验收」段里写的意图 bar**，而非某个普适数字。

---

## GP1. grind 墙 / 趣味密度不足（G1）

**Ask yourself**：玩家在抵达下一个趣味 beat（抉择 / 新剧情 / 新机制 / 资源拐点）前，要重复多少次无新信息的同一操作？

**形状**：长段相同低信息操作堆在一起（如靠重复同一"修炼 / 打工"动作几十次才够到下一次突破 / 抉择）；趣味 beat 被大量 filler 隔开，密度低到"像在工作"；早期 onboarding 期就先撞上重复墙，玩家还没尝到核心乐趣就要先肝。

## GP2. 每决策交互税过高（G2）

**Ask yourself**：完成一个核心决策，玩家要点几次、读多少字？其中多少是决策必需、多少是可折叠噪声？

**形状**：一个有意义的行动被过多点击 / 确认 / 必读文本稀释；反馈链冗长（多段叙事 + 多个气泡 + 顶栏刷新）让单次决策"手感"变重；每次决策都要重复一段不可跳过、不可加速的演出。

## GP3. 核心内容埋深 / 翻屏才可见（G2）

**Ask yourself**：玩家最想要的核心内容 / 入口，要滚动、翻屏或穿过多层导航才看得到吗？首屏直达的是核心还是边角？

**形状**：核心入口藏在下滑或多层 tab / 弹窗之后；首屏被次要信息占据、核心行动按钮在 fold 之下（默认视口看不到，逼用户先滚动才发现）；地图 / 列表要滚很久才找到目标项。
